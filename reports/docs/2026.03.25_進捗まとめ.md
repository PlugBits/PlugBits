## 2. 今回のUX改善で実装したこと

### 目的
「印刷ボタン押下後に同期的にPDF完成待ちするUX」をやめて、
「ジョブ起票 → 状態確認 → 完了後にPDF取得」の非同期UXへ寄せること。

### 実装した内容

#### Worker
変更ファイル:
- `worker/src/index.ts`

主な変更:
- `POST /render-jobs`
  - 即時に `jobId` と正規化済み status を返すように変更
- `GET /render-jobs/:id`
  - UI 向け payload を返すように変更
- `GET /render-jobs/latest?...`
  - 直近 job を取得する API を追加
- 内部状態を UI 向けに正規化
  - 内部: `leased / dispatched / processing / failed`
  - 公開: `queued / running / done / error`
- `pdfUrl`、`executionName`、`createdAt / startedAt / finishedAt` を公開 payload に含める
- reuse 判定を追加
  - 同一 `tenantKey + templateId + recordId + mode`
  - status が `queued / leased / dispatched / processing / running`
  - であれば既存 job を返す

追加ログ:
- `[RENDER_JOB_ACCEPTED]`
- `[RENDER_JOB_EXECUTION_REQUESTED]`
- `[RENDER_JOB_STARTED]`
- `[RENDER_JOB_DONE]`
- `[RENDER_JOB_ERROR]`
- `[RENDER_JOB_REUSED]`
- `[RENDER_JOB_LATEST_FOUND]`
- `[RENDER_JOB_LATEST_MISS]`

#### Plugin
変更ファイル:
- `plugin/src/renderJobs.ts`
- `plugin/src/desktop/index.ts`
- `plugin/src/mobile/index.ts`

主な変更:
- plugin client を整理
  - `createRenderJob`
  - `fetchRenderJobStatus`
  - `waitForRenderJob`
  - `buildRenderJobPdfUrl`
  - `fetchLatestRenderJob`
- desktop / mobile の印刷ボタンを非同期UX化
- 印刷押下後:
  - job 起票
  - status card 表示
  - polling で状態更新
- done 時:
  - `PDFを開く`
- error 時:
  - `再試行`

## 3. 実装後に発生した問題と対応

### 問題1: 画面更新後に「確認中...」のままになる
最初は latest 復元を入れたが、`確認中...` から戻らないケースが発生。

当初の見立て:
- latest API の timeout / finally 漏れ
- `job: null` 時の正常終了漏れ
- polling 開始失敗時の状態解除漏れ

### 問題2: 実際の本命原因は kintone イベント制約違反だった
Console error から原因が判明。

原因:
1. `app.record.detail.show` の登録を `DOMContentLoaded` 後に行っていた  
   → kintone の「同期登録必須」に反していた
2. restore 経路で `kintone.app.record.get()` / `kintone.mobile.app.record.get()` を呼んでいた  
   → detail.show 処理中に呼べずエラーになっていた

修正内容:
- `kintone.events.on(...)` をファイル読み込み時の同期登録に戻した
- `resumePrintJob()` / `resumeMobilePrintJob()` から current record 再取得を削除
- detail.show の `event.record` / `event.recordId` / `appId` を引数で渡す形へ変更
- restore 中例外が起きても catch して `印刷` に戻すよう修正

追加ログ:
- `[RENDER_JOB_DETAIL_SHOW_BEGIN]`
- `[RENDER_JOB_DETAIL_SHOW_ERROR]`
- `[RENDER_JOB_RESTORE_BEGIN]`
- `[RENDER_JOB_RESTORE_DONE]`

## 4. 現在の実際の挙動

現在の挙動:
1. 印刷ボタンを押す
2. `生成中` になる
3. 画面更新する
4. `印刷` ボタンに戻る
5. 再度押すと `確認中...` になる
6. 既存 job を拾う
7. 生成完了後に PDF を開けるようになる
8. 実際の挙動は「開く」というよりダウンロード寄り

### この挙動の評価
完全ではないが、以下はできている:
- サーバー側 reuse は効いている
- 二重起票はかなり防げている
- 再押下で既存 job に追従できる
- 完了後の PDF 取得導線はある

ただし未完成な点:
- record detail 表示時の自動 restore はうまく機能していない
- 画面更新後に自然に status card へ戻らない
- 再押下が必要になる

つまり今は、
**「自動復元」ではなく「再押下時の再接続」**
になっている。

## 5. 今回の結論

### 技術的な結論
Worker / Cloud Run Job の基盤は概ね成立している。  
問題の中心は、kintone detail 画面上で非同期 job の状態を気持ちよく見せ続ける UX 側にある。

### UX的な結論
detail 画面だけで
- 起票
- 状態追跡
- 完了
- 再取得
を全部綺麗にやるのは相性が悪い。

理由:
- 画面更新に弱い
- kintone イベント制約が厳しい
- 一時UIの寿命が短い
- 非同期処理の「居場所」がない

## 6. 今後の大きな方針

### Launcher連携案について
議論した結果、方針は以下。

#### 結論
- **商品としては別**
- **UXとしては連携可能**

整理:
- `PlugBits Reports`
  - 単体で動く
  - 拡張機能なしでも成立する必要がある
- `PlugBits Launcher`
  - 導入できる企業では UX を強化する
  - サイドパネルで進捗・履歴・完了通知を見せるのに向いている

つまり:
- Reports が主商品
- Launcher は体験ブースター

### なぜ完全依存にしないか
拡張機能を入れられない企業があるため。
もし Reports の使い勝手を Launcher 依存にすると、
- 導入障壁が上がる
- 単体商品として弱くなる
- 説明が複雑になる

したがって、**単体UXを先に固める** のが正しい。

## 7. 単体UXの方向性

### フィールド追加案
元アプリに
- 状態
- 最終出力日時
- PDFリンク
などのフィールドを持たせる案も出たが、これは重いと判断。

理由:
- ユーザー目線で業務アプリが汚れる
- 帳票の都合でフィールド追加が必要になる
- 導入時説明が増える

### 採用寄りの案
**kintone標準のカスタムビュー（HTML/JSを置けるやつ）を使い、Reports 自前の生成一覧を表示する。**

これはかなり有力。

---

## 8. 単体UXの採用方針（現時点）

### 基本コンセプト
**「レコードから依頼して、生成一覧で確認する」**

detail 画面では深追いしない。

### record detail 側
- ボタンは `PDFを作成` ないし `印刷用PDFを作成`
- 押下後:
  - 即 toast
  - `バックグラウンドで作成を開始しました`
  - `生成一覧で確認できます`

### 生成一覧側
kintone 標準のカスタムビューに、Reports が自前で生成一覧を描画する。

表示イメージ:
- 作成日時
- 帳票名
- 元レコード名
- ステータス
- 作成者
- ダウンロード / 開く
- 再試行

状態:
- 生成待ち
- 生成中
- 完了
- 失敗

### 保持方針
- 表示は最近 30 件まで
- まずは「表示30件」でよい
- 内部保持は将来拡張可能

### この案の利点
- 元アプリにフィールド追加不要
- 拡張機能なしで成立
- 非同期処理と相性が良い
- 後で Launcher サイドパネルにも同じ概念を流用できる

---

## 9. 単体UXの最終イメージ

### 短期
まずは Reports 単体で最低限成立する形を作る。

目指す形:
- detail 画面では起票だけ
- 状態確認は生成一覧
- 完了後にダウンロード / 開く
- 失敗時に再試行

### 中期
Launcher あり環境では、同じジョブ一覧をサイドパネルにも出す。

つまり:
- 単体版の「生成一覧」が土台
- Launcher 連携版はその上位UX

この構造がきれい。

## 11. 一言まとめ

今回の実装で、
- Cloud Run Job 基盤は通った
- 非同期 job 化はできた
- reuse もできている

ただし、
**detail 画面だけで非同期UXを完結させるのは相性が悪い**
ということがわかった。

今後の方針としては、
**Reports 単体で自前の生成一覧ビューを持ち、そこを単体UXの中心にする**
のが最も筋が良い。
